Nina Production SDK & API Key Guide

Drop a single script tag on your site and let Nina handle navigation, queries, and form-filling against your pages.

1. Backend URL & SDK script

After deploying to Railway, your backend has a public URL, e.g.: https://YOUR_BACKEND_URL. The SDK script is hosted at /nina-sdk.js on that backend.

Use this script tag on any page where you want Nina:

<!-- Replace with your actual Railway backend URL and API key -->
<script
  src="https://YOUR_BACKEND_URL/nina-sdk.js"
  data-nina-key="YOUR_NINA_API_KEY"
  data-nina-api="https://YOUR_BACKEND_URL"
  data-nina-position="bottom-right"
  async
></script>

If you omit data-nina-api, the SDK will call whatever origin it was loaded from (the host in the src attribute).

2. Getting a Nina API key

Backend Customers Per-site key

Nina identifies each website via a customer record. Use your backend to create a customer and retrieve its API key: 

POST /customers
Content-Type: application/json

{
  "name":  "My Demo Store",
  "email": "owner@example.com"
}

The response includes a generated API key. That key goes into the data-nina-key attribute in the script tag. Treat it like any other per-site public key: it identifies your tenant but does not grant admin access to Nina itself.

3. Required environment variables

On Railway (or any server), set these for the backend:

  • SUPABASE_URL – your Supabase project URL (NFS + memory store).
  • SUPABASE_KEY – service role key used by the backend only.
  • GEMINI_API_KEY – Google Gemini key used by gemini_client.py.
  • USE_PLAYWRIGHT – 0 for client‑only/browser execution, 1 to enable Playwright on the backend.

The SDK never sees these values. They are only read by the FastAPI backend and Supabase/Gemini clients.

4. Security notes

  • Your Nina API key is scoped to your site; it is safe to embed in the script tag, similar to public analytics or chat widget keys.
  • Do not expose your SUPABASE_KEY or GEMINI_API_KEY to the browser – keep them in server-side environment variables only.
  • If you rotate a Nina API key, just update the value in your script tag; no code changes required.

End-to-end example

The SDK handles UI, session memory, and the browser-side execution of navigate, fill_form, and click steps returned by your Nina backend.

Minimal HTML page

<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <title>My Store with Nina</title>
  </head>
  <body>
    <h1>Welcome to My Store</h1>
    <p>Browse products and ask Nina to help you navigate.</p>

    <!-- Nina SDK: replace with your backend URL + API key -->
    <script
      src="https://YOUR_BACKEND_URL/nina-sdk.js"
      data-nina-key="YOUR_NINA_API_KEY"
      data-nina-api="https://YOUR_BACKEND_URL"
      data-nina-position="bottom-right"
      async
    ></script>
  </body>
</html>

Customizing per environment

You can keep the same HTML and swap only the backend URL via a small server-side template or build-time replacement: 

<script
  src="{{ NINA_BACKEND_URL }}/nina-sdk.js"
  data-nina-key="{{ NINA_API_KEY }}"
  data-nina-api="{{ NINA_BACKEND_URL }}"
  async
></script>

For local development you can point both values to http://localhost:8000. In production use your Railway URL or custom domain.

How the SDK talks to your backend

When a user types a command, the SDK sends a POST request to your backend’s /run endpoint: 

POST ${API_BASE}/run
Content-Type: application/json

{
  "user_input":   "<what the user typed>",
  "base_url":     window.location.origin,
  "api_key":      "<data-nina-key>",
  "session_id":   "sess_...",
  "extra_params": {}
}

The backend:

  • Parses intent with Gemini (intent_parser.py).
  • Resolves actions using the NFS tree and Supabase.
  • Either returns navigate / fill_form steps for the SDK to execute, or an answer for query-style intents.

The SDK already knows how to handle needs_login, execute_api, and navigate results – you don’t need to write any extra browser code to use them.
